### 1) 지도 기반 점포·이벤트 통합 검색 및 탐색 UI 설계

점포는 좌표를 보유하지만 이벤트는 주소만 존재하여 지도에 즉시 표시할 수 없었습니다. 지도 UI 특성상 확대/축소/이동이 반복되면서 검색 결과와 UI 상태가 쉽게 어긋나 탐색 경험이 끊길 수 있었습니다.

점포와 이벤트를 서로 다른 화면/흐름으로 분리해 구현하는 방식도 검토했으나, 사용자가 “검색 → 표시 → 상세 → 재검색” 과정에서 맥락이 끊길 가능성이 높다고 판단했습니다.

점포/이벤트를 동일한 탐색 단위로 다루도록 설계를 전환하고, 지도 상의 동작(마커 표시, 오버레이 노출, 리스트 연동, 상세 보기)을 하나의 탐색 흐름으로 통합했습니다. 이벤트는 주소 기반 데이터라는 제약을 고려해, 지도 표시 이전 단계에서 좌표 변환을 수행하고 변환 완료된 데이터만 탐색 대상으로 사용하도록 구성했습니다.

사용자는 데이터 유형(점포/이벤트)을 의식하지 않아도 동일한 방식으로 검색·탐색할 수 있게 되었고, UI 흐름이 끊기지 않는 탐색 경험을 제공할 수 있었습니다.

### 2) 지도 상태 기반 검색 조건 설계 및 사용자 탐색 맥락 유지

지도 기반 서비스에서 검색은 키워드만으로 설명되지 않고, 사용자가 “현재 보고 있는 위치/화면 범위”가 곧 검색 조건이 됩니다. 페이지 이동/새로고침 시 검색 조건이 초기화되면 탐색 맥락이 단절되어 재입력 비용이 발생했습니다.

단순히 모든 상태를 한 곳(localStorage)으로 저장하는 방식은 구현이 쉽지만, 의도치 않은 재검색/중복 실행이 발생할 수 있어 적절하지 않다고 판단했습니다.

검색 요청 조건을 키워드 + 지도 중심 좌표 + 확대 레벨 + 반경으로 구성해, 동일 키워드라도 지도 상태에 따라 결과가 달라지도록 설계했습니다. “유지되어야 하는 탐색 조건”은 로컬 스토리지에 저장하고, “일회성 이동/처리에 필요한 값”은 세션 스토리지로 분리하여 상태를 관리했습니다.

탐색 맥락은 유지하면서도 불필요한 반복 실행을 방지할 수 있었고, 사용자의 재입력 비용을 줄여 탐색 UX를 개선했습니다.

### 3) 위치 기반 점포 조회 정확도 개선(행정구역 코드 기반 → 거리 기반 Top-N)

초기에는 행정구역(구 단위) prefix 조회로 점포를 불러왔으나, 구 경계 지역에서 가까운 점포가 누락되는 구조적 한계가 있었습니다.

클라이언트에서 후보 데이터를 넓게 가져온 뒤 거리 계산/필터링하는 방식을 검토했으나, 데이터량이 증가할수록 네트워크·렌더링·연산 비용이 커질 수 있어 비효율적이라고 판단했습니다.

DB에서 좌표 거리(ACOS 기반) 계산으로 정렬 후 Top-N(가까운 10개)만 조회하도록 전환했습니다. 응답 단계에서 가까운 점포만 내려주도록 설계하여 프론트의 후처리 부담을 줄였습니다.

경계 지역에서도 근접 점포 노출이 안정화되었고, 프론트 후처리가 줄어 응답성과 유지보수성이 함께 개선되었습니다.

### 4) 지도 인터랙션 충돌을 고려한 UI 동작 제어

지도 UI는 마커/오버레이/리스트/사이드바 등 요소 간 상호작용이 많아, 특정 UI 요소가 지도 이동/확대/축소를 방해할 경우 UX가 크게 저하될 수 있었습니다.

오버레이를 항상 노출시키는 방식은 정보 전달에는 유리하지만, 검색 결과가 많을 때 화면이 복잡해지고 지도 조작을 방해할 수 있어 적합하지 않았습니다.

오버레이는 “정보 전달”에 집중하도록 하고 지도 기본 인터랙션이 우선되도록 UI 동작을 제어했습니다.

확대 레벨에 따라 오버레이 노출 여부를 조정하고, 검색 결과 마커 분포를 기준으로 적절한 확대 레벨을 자동 조정하도록 구성했습니다.

용자가 별도 조작 없이도 검색 결과를 한눈에 파악할 수 있었고, 지도 조작이 자연스럽게 이어지는 탐색 경험을 제공할 수 있었습니다.

### 1) OpenAI API 기반 운동 루틴 추천 및 API 요청 로그 분석 시스템

유료 API 요청이 누적되는데도 요청별 비용(토큰 사용량)과 오류 원인을 체계적으로 추적할 수 없었습니다. 모델/버전 변경에 따른 응답 구조 차이가 발생해, 운영 단계에서 장애 원인 파악이 어렵다고 판단했습니다.

단순 호출 기능만 구현한 뒤 장애 발생 시마다 수동으로 로그를 확인하는 방식은, 운영/과금 정책 설계에 필요한 근거가 부족해 지속 가능하지 않았습니다.

요청/응답, 사용 모델, 입력·출력 토큰 수, 상태값을 구조화하여 저장하는 로그 분석 시스템을 설계·구현했습니다. 버전별 응답 차이를 고려해 파싱 로직을 분기 처리하고, 비정상 응답(잘못된 운동 ID, 분할 수 불일치 등)을 자동 감지하도록 구성했습니다.

사용자별 토큰 사용량을 기반으로 실제 비용을 수치로 확인할 수 있었고, AI 응답 오류를 빠르게 추적할 수 있는 운영 기반을 확보했습니다.

### 2) AI 추천 결과의 DB 무결성 보장(운동명 불일치 해결)

AI 추천 루틴은 DB에 존재하는 운동 데이터로만 구성되어야 했으나, 운동명 문자열 기반 응답은 DB와 불일치하여 저장 및 상세 매핑이 불가능한 케이스가 발생했습니다.

문자열 유사도 검증 후 재요청 방식은 비용/지연이 증가하고 재요청이 반복될 수 있어 비효율적이라 판단했습니다. 모든 운동명을 프롬프트에 넣는 방식도 검토했지만, 불일치가 완전히 제거되지는 않았습니다.

운동 ID(pt_idx) 기반 응답”으로 설계를 전환하고, 서버에서 응답 ID를 검증한 뒤 DB 매핑으로 화면/저장을 일관화했습니다. 유효하지 않은 ID나 누락 값은 로깅하여 원인 분석 및 품질 개선이 가능하도록 구성했습니다.

AI 추천 결과가 기록/통계/상세 연결 파이프라인에서 안정적으로 재사용 가능해졌고, 데이터 무결성을 기반으로 추천 기능의 신뢰도를 확보했습니다.

### 3) 구독·정기결제 시스템 및 정기결제 스케줄링

로컬 개발 환경 특성상 Webhook 사용이 불가능해 결제 결과를 자동 처리하기 어려웠고, 단순 결제 요청만으로는 실패/중복/누락 처리에 취약했습니다. 외부 Webhook 의존은 개발 환경 제약상 해결책이 될 수 없어, 내부에서 결제 상태를 추적/동기화할 수 있는 구조가 필요했습니다.

BillingKey 기반 정기결제 스케줄링 및 상태 모니터링 시스템을 설계·구현했습니다. 결제 예약 정보를 DB에 저장하고, 스케줄러로 주기적 상태 조회 후 변경 시에만 주문 상태를 업데이트했습니다. 호출 횟수 제한, 중복 처리 제어 로직을 적용하고 결제 성공 시 다음 결제 예약까지 자동화해 정기결제 사이클을 완성했습니다.

개발 환경에서도 안정적으로 정기결제 흐름을 검증할 수 있었고, 구독 서비스 운영을 고려한 결제 구조를 구축할 수 있었습니다.

### WBS 도입 후 무엇이 실제로 달라졌는지

기술 협업 과정에서 팀원끼리 "그 기능 언제 구현 되요?", "아, 금방 되요" 와 같은 주먹구구식 일처리 했음.  
그러나 WBS 도입 이후 일주일 단위로 저번주엔 어떤 작업을 했었고 이번주엔 무엇을 할 것이다. 라는 계획이 생김

### PortOne 스케줄러 처리 흐름

PortOne 스케줄러 처리 흐름은 아래처럼 설명하면 됩니다.

1. 예약 결제 생성 단계
2. 결제 예약 요청 시 `PaymentServiceImple.scheduleBillingKey`가 PortOne에 스케줄을 먼저 생성합니다.
3. 성공하면 `schedule_id`를 추출해 내부 `payment_order`에 `order_type='SCHEDULE'`, `order_status='READY'`로 저장합니다.
4. 이때 DB와 PortOne 상태를 맞추기 위해 DB 저장 실패 시 PortOne 예약 취소 보상 로직이 들어가 있습니다.

5. 일일 모니터 배치 시작
6. `ScheduledPaymentMonitor.processDailyPaymentBatch()`가 매일 `00:10`(KST 기준 의도) 실행됩니다.
7. `payment.monitor.enabled=false`면 즉시 종료하여 특정 서버(마스터)만 실행하게 설계되어 있습니다.
8. 당일 `00:00:00~23:59:59` 범위의 `READY + SCHEDULE + schedule_id not null` 주문을 조회합니다.

9. 주문별 PortOne 상태 조회
10. 각 주문의 `schedule_id`로 `GET /payment-schedules/{scheduleId}?storeId=...`를 호출합니다.
11. 응답 `status`를 내부 상태로 매핑합니다.
12. 매핑 규칙:
	- `SUCCEEDED -> PAID`
	- `FAILED -> FAILED`
	- `REVOKED -> CANCELLED`
	- `SCHEDULED/PENDING -> UNCHANGED(대기 유지)`
13. 상태 반영(동시성/멱등성 핵심)
14. 상태 변경이 필요할 때만 트랜잭션 메서드 `updateOrderStatusInTransaction`을 실행합니다.
15. 업데이트 전 `order_idx` 재조회로 현재 상태를 확인하고, 이미 `READY`가 아니면 반영하지 않습니다.
16. 실제 SQL도 `WHERE order_status='READY'` 조건부 업데이트라 중복 반영을 방지합니다.
17. `PAID`일 경우 `order_paydate`와 PortOne 실제 결제 ID(`payments[0].id`)를 저장합니다.

18. 성공 후 후속 처리(자동 다음 달 예약)
19. 결제가 `PAID`이고 주문 타입이 `SCHEDULE`이며 `payment.auto.schedule.enabled=true`면 다음 달 자동 예약을 시도합니다.
20. 먼저 회원의 기존 예약(`READY`) 존재 여부를 확인해 중복 예약을 막습니다.
21. 없으면 `scheduleNextMonthPayment`로 31일 후 결제를 재예약해 구독 사이클을 이어갑니다.

22. 안정성 장치
- API 타임아웃/네트워크 예외를 주문 단위로 격리 처리해 전체 배치 중단을 피함
- API 호출 카운터 제한(`maxApiCallsPerMinute`) 및 배치 간격 sleep으로 외부 API 부하 완화
- 배치 종료 시 성공/실패/대기/건너뜀 건수를 집계 로그로 남김

한 줄 요약:
“Webhook 제약 환경에서, 일 배치 Polling으로 PortOne 예약 상태를 내부 주문 상태와 동기화하고, `READY` 조건부 업데이트로 멱등성을 보장하며, 성공 시 다음 달 예약까지 자동 연결하는 구조”입니다.

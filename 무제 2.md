1) StoreMap(스토어맵) : 지역축제 및 길거리 음식 정보 제공 서비스

기술 스택 : Spring, MyBatis, JSP, Java11, Javascript, Oracle


1-1) 지도 기반 점포·이벤트 통합 검색 UI 구현

- 점포(좌표 보유)와 이벤트(주소 중심) 데이터를 각각 다른 화면/흐름으로 분리하는 방식도 검토했으나, 사용자가 검색 → 표시 → 상세 → 재탐색 과정에서 맥락이 끊기는 문제가 있었습니다.
    
- 이에 지도 중심/줌레벨/반경을 공통 검색 조건으로 정의하고, 좌표/주소 기반 데이터를 동일 탐색 흐름으로 통합하여 “한 번의 검색 조건으로 계속 탐색 가능한 UI”를 설계했습니다.
    
- 결과적으로 데이터 유형이 달라도 탐색 방식이 일관되어 사용자 혼란을 줄였고, 화면/기능이 분리될 때 발생하는 중복 구현(검색/페이징/상세연결)을 최소화해 유지보수성을 높였습니다.
    

1-2) 검색 자동완성 및 탐색 맥락 유지 기능 구현

- 자동완성을 단순 텍스트 추천으로만 제공하면 사용자가 다시 검색 조건을 조립해야 해 이탈 가능성이 높다고 판단했습니다.
    
- 점포·이벤트를 동시에 제안하는 자동완성과 키보드 탐색을 구현하고, 검색 조건을 세션/로컬 스토리지로 분리 저장해 페이지 이동/새로고침 후에도 이전 탐색 맥락이 유지되도록 설계했습니다.
    
- 그 결과 반복 입력 비용을 줄여 탐색 속도를 개선했고, 팀 관점에서는 “검색 조건 관리” 로직을 표준화하여 유사 기능 확장 시 재사용 가능하게 만들었습니다.

1-3) 위치 기반 점포 조회 정확도 개선(행정구역 코드 기반 → 좌표 거리 기반 Top-N 조회)

- 초기에는 행정구역(구 단위) 코드 prefix로 점포를 조회했으나, 구 경계 지역에서 가까운 점포가 누락되는 구조적 한계가 있었습니다.
    
- 클라이언트에서 후보를 넓게 가져와 거리 계산/필터링하는 방식도 검토했지만, 데이터량 증가에 따른 네트워크·렌더링·연산 부담이 커질 수 있다고 판단했습니다.
    
- 이에 DB에서 좌표 거리(ACOS 기반) 계산으로 정렬 후 Top-N(가까운 10개)만 조회하도록 전환하여, 경계 지역에서도 근접 점포 노출을 안정화했습니다.
    
- 추가로 MyBatis LIKE 조건을 포함 매칭에서 prefix 매칭으로 수정해 조회 정확도를 보완했고, 결과적으로 클라이언트 후처리를 줄여 응답 부담과 UX를 함께 개선했습니다.

		- 그럼에도 불구하고 운동명 불일치 케이스가 많았음 (성공률 약 80%)
	- 운동명과 idx를 매핑하여 프롬프트에 다 명시, 응답은 idx로 (이는 당연히 성공률 100%)

- 이에 요청/응답/토큰/에러를 구조화해 저장하고, 버전 비교·통계·상세 조회가 가능한 로그 분석 체계를 설계·구현했습니다.
    
- 결과적으로 구독 요금 산정을 위한 비용 근거를 확보했고, 오류 재현 및 원인 분석 시간이 단축되어 서비스 안정성 측면에서 팀의 운영 부담을 줄였습니다.
    

2-2) 구독·정기결제 시스템 및 정기결제 스케줄링 구현

- 그 결과 결제 처리의 신뢰성을 높였고, 운영 시 이슈 발생 시점과 원인 파악이 쉬워져 팀 차원의 장애 대응 품질이 개선되었습니다.

# 프로젝트에서 팀장으로서의 기여도

일단 내가 실제로 팀장으로서 결정했던 것들과 기여도들을 자유롭게 적을게. 너가 알아서 적절하게 첨삭해줘.

## StoreMap

- 팀원 성향을 고려한 업무 분담 지시
- 노션을 통한 협업 관리 워크페이스 구축
- 회의록 작성
- PPT 제작 (이건 개발 외 내용이라 적어야할지 잘 모르겠음)

## FitSync

- StoreMap에서 언급한 것들을 기본적으로 다 수행
- 의사결정 리딩 (일단 생각 나는 굵직한거 먼저 작성함)
	- 운동기록과 운동루틴의 관계
		- 나만의 운동루틴(routine)을 기반으로 운동기록(record) 만들 수 있었음
		- 운동기록과 운동루틴의 연결성이 있다면 좋을거 같다고 팀원이 제안 (운동기록을 볼때 이 기록이 어떤 루틴 기반으로 만들어졌는가를 알 수 있게)
		- 운동기록과 운동루틴 사이에 fk를 걸 경우 아래와 같은 문제가 발생
		- 루틴을 수정할 경우 운동기록에 영향을 줌
		- 방법은 존재했음 (예를 들어 자체적인 도메인 버저닝을 통해 기록) 그러나 처음 의도와 너무 기능 부담이 커짐
		- 원래 우리가 의도한 도메인이 뭐였는지 회의를 통해 고민함
		- 루틴은 "간편 자동완성"과 같은 개념임.
		- 그러므로 우리는 루틴과 기록 사이에 연결점을 두지 않는 것이 맞다고 판단함.
- 확실한 일정 조율을 위해 주간 업무 계획서를 작성 및 매주 월요일에 점검
	- StoreMap 당시 일정 조율 실패로 인해 몇몇 기능들을 미구현함 (팀원이)

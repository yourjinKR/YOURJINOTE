
## 추가 학습내용

### EntityGraph 사용

```java
@Getter  
@Entity  
@Table(name = "body_detail_parts")  
@NoArgsConstructor
public class BodyDetailPart {

	@ManyToOne(fetch = FetchType.LAZY)  
	@JoinColumn(name = "body_part_id", nullable = false)  
	private BodyPart bodyPart;

}
```

`BodyDetailPart`와 `BodyPart`는 N:1 관계이다.  
지연 로딩이 기본값이기에 `BodyPart`까지 즉시 데이터를 가져온다면 N+1문제가 발생한다.  

이때 `@EntityGraph`를 사용하여 간단히 조회 쿼리를 구현했다.

```java
@EntityGraph(attributePaths = "bodyPart")  
List<BodyDetailPart> findAllByOrderByIdAsc();
```

로그와 같이 JOIN 절을 통해 데이터를 가져옴을 확인.

#### 주의사항

- 지금과 같은 `@ManyToOne` 관계예서는 괜찮으나 `@OneToMany` 관계에서는 사용시 주의
- JOIN하는 엔티티의 크기가 크다면 사용시 주의

#### 체크리스트

- **이 연관을 80~100% 확률로 쓸 건가?**
    - Yes → 같이 가져와도 됨
    - No → LAZY 유지 + 필요 시 별도 조회/DTO~~
        
- **연관이 ToOne(ManyToOne/OneToOne)인가?**
    - Yes → EntityGraph 효과 좋고 부작용 적음
    - No(ToMany) → 조인 뻥튀기/페이징 이슈 가능, 더 신중
        
- **목록 + 페이징인가?**
    - Yes → ToMany 그래프는 특히 피하기 (대체전략 고려)
        
- **응답이 DTO고 필요한 컬럼이 적은가?**
    - Yes → DTO 프로젝션이 더 효율적일 가능성 큼~~


### JPQL이 더 fit하다?

```java
@Query("""
	select new app.fitsync.domain.exercise.dto.body.BodyDetailPartListResponse(
		d.id, d.name, p.name
	)
	from BodyDetailPart d
	join d.bodyPart p
	order by d.id asc
""")
List<BodyDetailPartListResponse> findAllListResponses();
```

이후 JPQL로 리팩토링했다.  
JPQL의 단점이라면 `BodyPart`와 `BodyDetailPart`를 가져오는 기능이 추가될때 재활용하지 못한다는 점이지만 지금과 같은 상황은 괜찮다고 판단.

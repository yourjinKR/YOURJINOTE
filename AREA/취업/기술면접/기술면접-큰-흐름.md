
# 🔹 백엔드 개발자 필수 기술 스택 – 큰 흐름 질문 & 답변


## 1️⃣ OOP (객체지향)

### Q1. 객체지향이 등장한 근본적인 이유는 무엇인가요?

**A.**  
절차지향의 전역 상태 공유와 강한 결합 문제를 해결하고,  
현실 세계의 개념을 코드로 모델링하여 **유지보수성과 확장성을 높이기 위해** 등장했습니다.


### Q2. 객체지향의 핵심 가치는 무엇인가요?

**A.**  
변경에 강한 구조를 만드는 것입니다.  
캡슐화로 변경 범위를 제한하고, 다형성으로 구현 교체를 가능하게 합니다.


### Q3. 상속보다 조합을 선호하는 이유는 무엇인가요?

**A.**  
상속은 강한 결합을 만들기 쉽고 변경에 취약합니다.  
조합은 런타임에 유연한 확장이 가능해 실무에서 더 안전한 선택입니다.


### Q4. 인터페이스는 왜 필요한가요?

**A.**  
구현이 아닌 **역할에 의존**하게 하여 결합도를 낮추고,  
다형성과 테스트 용이성을 확보하기 위해 필요합니다.


## 2️⃣ Java

### Q1. 자바의 가장 큰 특징은 무엇인가요?

**A.**  
JVM 기반으로 실행되어 **플랫폼 독립성**을 가지며,  
자동 메모리 관리와 강력한 객체지향 모델을 제공합니다.


### Q2. JVM이 왜 필요한가요?

**A.**  
운영체제별 차이를 JVM이 흡수하여 동일한 바이트코드를 실행할 수 있게 하고,  
GC를 통해 메모리 관리를 언어 차원에서 처리하기 위해 필요합니다.


### Q3. 자바가 서버 사이드 언어로 많이 쓰이는 이유는 무엇인가요?

**A.**  
안정성, 풍부한 표준 라이브러리, 강력한 멀티스레딩 지원,  
그리고 Spring 생태계 덕분에 대규모 서버 개발에 적합합니다.


### Q4. 컴파일 언어임에도 생산성이 높은 이유는 무엇인가요?

**A.**  
정적 타입 기반으로 컴파일 타임 오류를 줄이고,  
프레임워크와 도구 생태계가 개발 복잡성을 크게 줄여주기 때문입니다.


## 3️⃣ Spring

### Q1. Spring이 등장한 배경은 무엇인가요?

**A.**  
EJB 중심의 무거운 엔터프라이즈 개발의 복잡성과 결합도를 줄이기 위해 등장했습니다.


### Q2. Spring의 핵심 철학은 무엇인가요?

**A.**  
IoC와 DI를 통해 객체 생명주기와 의존성 관리를 프레임워크가 담당하고,  
개발자는 비즈니스 로직에 집중하도록 돕는 것입니다.



### Q3. DI가 없으면 어떤 문제가 있나요?

**A.**  
객체 생성과 사용이 강하게 결합되어  
테스트가 어렵고 구현 교체가 힘들어집니다.



### Q4. 왜 Controller–Service–Repository 구조를 쓰나요?

**A.**  
역할 분리를 통해 변경 영향 범위를 줄이고  
비즈니스 로직을 명확히 보호하기 위함입니다.

## 4️⃣ JavaScript

### Q1. JavaScript는 왜 웹에서 필수가 되었나요?

**A.**  
브라우저에서 실행되는 유일한 표준 스크립트 언어로,  
사용자 인터랙션과 동적 UI 처리를 담당하기 때문입니다.



### Q2. JS가 단일 스레드인데도 비동기가 가능한 이유는 무엇인가요?

**A.**  
이벤트 루프와 비동기 콜백 구조로  
블로킹 없이 작업을 처리할 수 있기 때문입니다.



### Q3. JS가 서버 사이드(Node.js)까지 확장된 이유는?

**A.**  
비동기 I/O에 강하고,  
프론트와 백엔드 언어를 통일해 개발 생산성을 높일 수 있기 때문입니다.


## 5️⃣ React

### Q1. React가 등장한 배경은 무엇인가요?

**A.**  
DOM 직접 조작의 복잡성과 상태 관리 문제를 해결하기 위해 등장했습니다.



### Q2. React의 핵심 개념은 무엇인가요?

**A.**  
UI를 상태의 함수로 바라보고,  
상태 변화에 따라 자동으로 화면을 갱신하는 선언형 UI 모델입니다.



### Q3. 컴포넌트 기반 설계의 장점은?

**A.**  
재사용성과 유지보수성이 높아지고,  
UI 구조를 논리적으로 분리할 수 있습니다.



### Q4. 백엔드 관점에서 React의 의미는?

**A.**  
백엔드는 순수 API 제공에 집중하고,  
프론트는 독립적으로 UI와 상태를 관리하는 역할 분리가 가능해집니다.



## 6️⃣ RDB (관계형 데이터베이스)

### Q1. RDB가 여전히 많이 쓰이는 이유는 무엇인가요?

**A.**  
정합성과 트랜잭션을 강하게 보장하며,  
복잡한 관계를 명확하게 표현할 수 있기 때문입니다.


### Q2. 정규화의 목적은 무엇인가요?

**A.**  
중복을 제거하여 데이터 무결성과 유지보수성을 높이기 위함입니다.



### Q3. 트랜잭션이 중요한 이유는?

**A.**  
여러 작업을 하나의 논리적 단위로 묶어  
데이터 일관성을 보장하기 위해 필요합니다.



## 7️⃣ Oracle

### Q1. Oracle이 엔터프라이즈 환경에서 많이 쓰이는 이유는?

**A.**  
강력한 트랜잭션 처리, 안정성, 정교한 권한 관리와 튜닝 기능 덕분입니다.



### Q2. Oracle의 시퀀스 개념은 왜 필요한가요?

**A.**  
동시성 환경에서 중복 없는 PK 생성을 보장하기 위해 필요합니다.



### Q3. Oracle이 표준 SQL 외 기능을 제공하는 이유는?

**A.**  
대규모 트래픽과 복잡한 비즈니스 요구를 처리하기 위한 성능 최적화 목적입니다.



## 8️⃣ HTTP

### Q1. HTTP는 어떤 역할을 하나요?

**A.**  
클라이언트와 서버 간의 **요청–응답 규칙을 정의한 프로토콜**입니다.



### Q2. HTTP가 무상태(stateless)인 이유는?

**A.**  
서버 확장성과 단순성을 확보하기 위해서이며,  
상태는 쿠키나 토큰으로 외부 관리합니다.



### Q3. REST가 등장한 이유는?

**A.**  
자원을 중심으로 한 일관된 API 설계를 통해  
확장성과 가독성을 높이기 위해서입니다.



## 9️⃣ 기타 웹 백엔드 기초

### Q1. 백엔드의 핵심 책임은 무엇인가요?

**A.**  
비즈니스 로직 처리, 데이터 관리, 보안, 트랜잭션 보장입니다.


### Q2. 왜 로그와 모니터링이 중요한가요?

**A.**  
문제 발생 시 원인 분석과 시스템 안정성을 확보하기 위해 필수입니다.



### Q3. 인증과 인가는 왜 분리되나요?

**A.**  
인증은 “누구인가”,  
인가는 “무엇을 할 수 있는가”로 책임이 다르기 때문입니다.

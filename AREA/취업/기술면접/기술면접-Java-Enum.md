# 다시 점검할 부분

### 1. Enum은 왜 `==` 비교가 안전한가?

#### 지뢰 질문

- Enum 상수는 왜 `equals()` 대신 `==` 비교가 권장될까?
- Enum 상수가 싱글톤처럼 동작한다고 말할 수 있는 근거는 무엇일까?
- 리플렉션이나 직렬화를 사용해도 Enum 인스턴스가 하나만 유지되는 이유는 무엇일까?

#### 핵심 연결

- JVM 클래스 로딩
- 싱글톤 보장
- 객체 동일성(identity)

---

### 2. `ordinal()`을 실무에서 사용하면 왜 위험한가?

#### 지뢰 질문

- `ordinal()` 값은 언제, 어떤 상황에서 깨질 수 있을까?
- Enum 상수의 선언 순서를 변경하면 어떤 문제가 발생할까?
- JPA나 MyBatis에서 Enum을 DB에 저장할 때 `ordinal` 기반 매핑이 왜 위험한 선택일까?

#### 핵심 연결

- 유지보수 리스크
- DB 스키마 안정성
- Enum 설계 원칙

---

### 3. Enum에 생성자가 private인 이유는 무엇인가?

#### 지뢰 질문

- Enum 생성자는 왜 명시적으로 `private`를 붙이지 않아도 private일까?
- 개발자가 Enum 인스턴스를 직접 생성하지 못하도록 막은 이유는 무엇일까?
- 이 제약이 없다면 어떤 설계 문제가 발생할 수 있을까?

#### 핵심 연결

- 객체 생성 통제
- JVM 차원의 Enum 보장
- 닫힌 집합 개념

---

### 4. Enum은 단순 상수 묶음이 아니라 왜 “타입”인가?

#### 지뢰 질문

- 상수 클래스(`public static final int`)와 Enum의 본질적인 차이는 무엇일까?
- Enum이 컴파일 타임 타입 체크를 제공한다는 말은 어떤 의미일까?
- Spring에서 Enum을 메서드 파라미터나 설정 값으로 사용할 때 얻는 이점은 무엇일까?

#### 핵심 연결

- 타입 안정성
- 컴파일 타임 검증
- Spring 설정 및 도메인 모델링

---

### 5. Enum + switch 설계는 언제 깨질 수 있는가?

#### 지뢰 질문

- Enum에 새로운 상수를 추가했을 때 switch 문은 컴파일 에러를 발생시킬까?
- 모든 case를 처리하지 않았을 경우 발생하는 문제는 무엇일까?
- 이 문제를 방지하기 위한 Enum 설계 방식에는 어떤 것들이 있을까?

#### 핵심 연결

- 분기 누락 버그
- 유지보수성
- Enum 기반 전략 패턴

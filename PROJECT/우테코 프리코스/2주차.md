
## 과제를 진행하며
### static을 너무 남발했는가
유틸적인 기능은 전부 `static`을 사용했고 과제를 마친 후 다시보니 `static`을 많이 사용한거 같습니다. static에 대해 검색해보니 장단점이 존재하지만 **"객체지향이지 못하다"** 글귀가 제 마음을 불편하게 만들었습니다.
> 각자만의 `static` 사용 기준과 이번 과제 중 저와 다르게 정적 필드, 메서드를 활용하신 분이 있으시다면 이에 대해 이야기 나눠봤으면 좋겠습니다. 
### 검증과정
검증과정을 어디서 수행해야 되는가에 고민이 많았습니다.
- 각각의 객체(`Car`,`Racing`)내부에서 책임져볼까?
- `validateXXXX()`와 같은 검증 메서드는 리턴값을 보통 `void`로 사용한다는데 어떻게 하지?
만약 그럴 경우 아래와 같이 수많은 Car 객체를 생성할 일이 있다면 비효율적이라는 생각이 들었습니다.
```
자동차1,자동차2,자동차3....자동차9999
```
제가 내린 결론은 `Validator`라는 검증과정을 수행하는 역할을 따로 분리하여 정적 메서드를 추가하고 `request`에 대한 전체적인 검증을 하도록 수행했습니다.
물론 현재 구조에서도 아래와 같은 구분자 파싱 행위를 2번 정도 반복하는 등 다소 아쉬운 점이 있습니다.
```java
String[] carNameArray = carNames.split(Delimiter.SYMBOL);
```
> 아래 2가지 정도에 대해서 이야기 해주실게 있으신 분들은 리뷰 남겨주시면 감사하겠습니다.
> 1. 검증과정을 어떤 과정 내에서 수행했는지
> 2. 응답값 뿐만 아니라 객체 생성 과정에서도 검증 과정을 수행해야 되는지

## 아쉬운 점
### DTO
`RacingRequest`와 `RacingResponse`로 DTO를 구현했으나. 
`RacingResponse`가 도메인에 다소 의존적이며 결국 데이터를 감싸는 역할만을 수행함을 뒤늦게 확인했습니다.

### 테스트 코드
테스트 코드에 아직 익숙치 않았기에 어느정도 완성 후에 테스트 코드를 작성했습니다.
그렇기에 "이런 것까지 테스트해봐야 하나?" 라는 의문을 시작으로 테스트 코드를 성실히 작성하진 않았습니다. 도메인과 예외처리에 대한 테스트 코드만 작성했습니다.
앞으로 테스트 코드 작성에 익숙해질 필요가 있을거 같습니다.

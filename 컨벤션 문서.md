# Git Flow

## Branch Naming

- `main`
    - 제품 출시 브랜치
    - 최종 배포용 브랜치
    - 항상 배포 가능한 안정 상태를 유지한다.
    - `main`에 반영된 코드는 운영 환경에 배포한다.
        
- `dev`
    - 다음 출시 버전을 개발하는 통합 브랜치
    - 기능 개발 브랜치들을 병합하여 통합 테스트를 진행한다.
    - 팀 내부 확인 및 중간 배포용 브랜치로 사용한다.
    - `dev`에 반영된 코드는 개발/스테이징 환경에 배포한다.
        
- `feat/*`
    - 기능 추가 브랜치
    - 새로운 기능 개발 시 `dev`에서 분기하여 생성한다.
    - 개발 완료 후 `dev`로 merge 한다.
        
- `fix/*`
    - 일반 버그 수정 브랜치
    - 개발 중 발견된 버그 수정 시 사용한다.
    - `dev`에서 분기하여 생성하고, 수정 후 `dev`로 merge 한다.
        
- `refactor/*`
    - 리팩토링 브랜치
    - 기능 변화 없이 구조 개선, 코드 정리, 중복 제거 등을 수행한다.
    - `dev`에서 분기하여 생성하고, 작업 후 `dev`로 merge 한다.
        
- `docs/*`
    - 문서 작업 브랜치
    - README, API 문서, 컨벤션 문서 등 문서 관련 수정 시 사용한다.
        
- `hotfix/*`
    - 운영 환경 긴급 수정 브랜치
    - 운영 배포 후 발생한 긴급 장애나 치명적 오류를 빠르게 수정할 때 사용한다.
    - `main`에서 분기하여 생성한다.
    - 수정 완료 후 `main`과 `dev` 양쪽에 반영한다.


> `main`과 `dev`를 제외한 작업 브랜치는 아래 예시와 같이 넘버링으로 간략화  
> (예시 : `feat/1`, `refactor/6`)


# Commit Message Convention

커밋 메시지만 보고도 어떤 작업인지 알 수 있도록 규칙을 정합니다.

**구조:** `type: description`

- **Feat**: 새로운 기능 추가
- **Fix**: 버그 수정
- **Docs**: 문서 수정 (README, Swagger 등)
- **Style**: 코드 포맷팅, 세미콜론 누락 등 (코드 변경 없음)
- **Refactor**: 코드 리팩토링
- **Test**: 테스트 코드 추가
- **Chore**: 빌드 업무, 패키지 매니저 설정 등
    
# PULL Request Template

```
### 📝 작업 내용
- [ ] 작업 1
- [ ] 작업 2

### 📸 스크린샷 (선택)
- UI 변경사항이 있다면 첨부해 주세요.

### 🔗 관련 Issue
- Resolved: #이슈번호

### ❓ 리뷰어에게 확인받고 싶은 점
- 이 부분의 로직이 최선일까요?
```

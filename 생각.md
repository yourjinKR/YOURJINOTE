# 1) 이력서용 트러블슈팅 (5~8줄 / STAR+RCA 압축)

위치 기반 점포 조회 정확도 개선(행정구역 코드 기반 → 좌표 기반 거리 계산)

- 카카오맵 dragend 이벤트로 지도 중심좌표를 받아 ‘구(區) 단위 행정구역 코드 prefix’로 점포를 조회했으나, 구 경계 지역에서 가까운 점포가 누락되는 문제가 발생했습니다.
    
- MyBatis LIKE 조건을 LIKE '%' || gu || '%'로 작성해 포함 매칭이 될 여지가 있어, LIKE gu || '%'로 변경하여 prefix 매칭 정확도를 높였습니다.
    
- 그러나 행정구역 단위 검색은 경계/반경 요구사항을 구조적으로 충족하기 어려워, DB에서 좌표 거리 기반으로 가까운 점포를 직접 추출하는 방식으로 전환했습니다.
    
- Oracle에 RADIANS/DEGREES 함수를 정의하고 ACOS 기반 거리 계산식을 적용해, 지도 중심 기준 가까운 점포 10개를 ROWNUM Top-N으로 조회했습니다.
    
- 그 결과, 경계 지역에서도 근접 점포 노출이 안정화되었고, 클라이언트 후처리(대량 후보 거리 계산) 대신 DB 정렬/필터링으로 응답 부담을 줄였습니다.
    

> (기술 키워드) Kakao Map dragend, 행정구역 코드 prefix, MyBatis LIKE 최적화, Oracle 함수, ACOS 거리계산, Top-N(ROWNUM)

# 2) 포트폴리오용 트러블슈팅 (1페이지 케이스 스터디)

## 위치 기반 점포 조회 정확도 개선

### 0. 배경

현위치 기반으로 점포를 보여주는 기능을 구현해야 했고, 처음에는 “좌표 → 행정구역(시/구/동) 변환”을 활용하면 간단히 해결될 것이라 판단했다.  
지도 이동(드래그)마다 중심 좌표를 얻고 해당 좌표의 행정구역을 저장한 뒤, 그 행정구역에 속한 점포를 조회하는 방식으로 시작했다.

---

### 1. 문제 상황

**문제:** ‘구(區)’ 단위로 점포를 조회하는 방식이 의도와 다르게 동작했다.

- 구현 목표는 “사용자가 현재 보고 있는 지도 중심 기준으로 **가까운 점포**를 보여주는 것”이었지만, 실제로는 “해당 **구에 속한 점포 전체**를 보여주는 기능”이 됐다.
    
- 특히 **구와 구 경계에 걸친 지역**에서 사용하면 가까운 점포가 다른 구에 속할 경우 누락되어 UX가 떨어질 가능성이 컸다.
    

---

### 2. 원인을 찾는 과정 (조사/검증)

1. **지도 중심 좌표 갱신 흐름 확인**
    

- `dragend` 이벤트에서 중심 좌표를 매번 갱신하고 행정구역을 로딩하는 흐름을 구성.
    
- 좌표 → 행정구역(코드) 매핑은 정상적으로 동작함.
    

2. **‘구 단위 조회’ 로직 검증**
    

- 행정구역 코드는 예: `11680xxxx` 형태로 ‘구’가 동일하면 prefix가 동일(강남구: `11680...`)
    
- 서비스 로직에서 `code / 100000`으로 구 코드(11680)를 추출하는 방식은 타당.
    

3. **MyBatis LIKE 사용 방식 점검**
    

- 초기 쿼리: `LIKE '%' || #{gu} || '%'`
    
    - 의도: “구 코드가 포함된 store_regcode 매칭”
        
    - 문제: `%11680%`는 prefix 매칭이 아니라 **문자열 포함 매칭**이라 데이터 형태에 따라 원치 않는 매칭 가능성이 있음.
        
- 수정 쿼리: `LIKE #{gu} || '%'`
    
    - prefix 매칭으로 “구 코드로 시작하는 데이터만” 정확히 조회되도록 수정.
        

4. **요구사항 적합성 재검토**
    

- 쿼리를 정확화해도, 본질적으로 “행정구역 단위 조회”는 “반경 기반 근접 점포 조회” 요구와 어긋남을 확인.
    
- 클라이언트에서 거리 계산 클래스를 만들어 “시 단위로 크게 가져온 뒤 거리로 필터”도 검토했으나, 후보가 많아질수록 **후처리 비용/응답 지연**이 커질 것으로 판단.
    

---

### 3. 원인(근본 원인) 정의

- **근본 원인:** “행정구역 코드 기반 조회(구 단위)는 경계 지역에서 근접 점포를 보장할 수 없는 구조적 한계”
    
- **부수 원인:** LIKE 조건을 포함 매칭으로 작성해 매칭 정확도가 떨어질 여지가 있었음.
    

---

### 4. 해결 접근 (선택지 비교)

- A안) 구 단위(또는 시 단위)로 가져온 뒤 **클라이언트에서 거리 계산 후 필터링**
    
    - 단점: 후보 데이터가 많아질수록 네트워크/렌더링/연산 비용 증가, UX 악화
        
- B안) DB에서 **거리 계산 + 정렬 + Top-N**으로 필요한 데이터만 내려줌 ✅ 채택
    
    - 장점: 서버/DB가 정렬과 필터를 담당 → 클라이언트 부담 감소, 결과 일관성↑
        
    - 결과: “지도 중심에서 가까운 점포”라는 요구를 직접 충족
        

---

### 5. 해결 방법 (구현)

1. **DB 함수 정의 (Oracle)**
    

- `RADIANS`, `DEGREES` 함수를 정의하여 거리 계산식에 사용.
    

2. **거리 기반 Top-N 조회**
    

- 구면코사인(ACOS) 기반 거리 계산식을 SQL에서 계산하여 `distance`를 구함.
    
- `ORDER BY distance` 후 `ROWNUM <= 10`으로 지도 중심 기준 가까운 10개 점포만 반환.
    

3. **프론트 적용 (지도/리스트/마커 동기화)**
    

- 검색 결과 콜백 `apply2map(data)`에서
    
    - 기존 마커 제거 → 새 마커 등록 → 리스트 렌더링 → 리스트-마커 매핑 → 지도에 마커 표시
        
- DOMContentLoaded 시점 이슈가 있는 변수는 재선언/재바인딩하여 안정화.
    

---

### 6. 결과 (효과)

- 구 경계 지역에서도 “가까운 점포”를 안정적으로 노출할 수 있게 되어 **요구사항에 부합하는 UX**를 확보.
    
- 클라이언트가 큰 후보군을 받아 거리 계산/필터링하는 방식 대비, DB에서 Top-N만 내려 받아 **응답 페이로드와 렌더링 부담을 감소**.
    
- 추가로 LIKE 조건을 prefix 매칭으로 수정하여, 행정구역 코드 기반 조회가 필요한 화면에서도 **매칭 정확도를 향상**.
    

---

### 7. 회고 / 다음 개선

- ACOS 기반 거리 계산은 전체 테이블에 대해 계산 비용이 생길 수 있으므로,
    
    - (가능한 개선) **Bounding Box(위경도 범위) 1차 필터 + 거리 정렬**로 최적화 고려
        
    - (가능한 개선) 공간 인덱스/Geo 기능(PostGIS 등) 도입 검토
        
- “정확도(근접)”와 “성능(계산 비용)” 사이의 트레이드오프를 설계로 해결한 경험으로 정리 가능.

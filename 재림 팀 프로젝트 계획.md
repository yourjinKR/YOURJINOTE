# GPT 피셜 체크리스트

## 0) 킥오프 전에 준비(1~2시간)

-  팀원 역할/가용시간 파악: 주당 가능 시간, 병행 일정, 마감일(학교/대회/공모전 등)
    
-  소통 채널 확정: 디스코드/슬랙/카톡 중 1개 + 공지용 채널 분리
    
-  작업 툴 확정: GitHub + Notion(or Wiki) + Figma(or FigJam)
    
-  회의 규칙: 회의 빈도/시간, 회의록 담당, 의사결정 방식(다수결/리더결정)
    

## 1) 주제 발굴 & 선정 체크리스트

### 1-1. 아이디어 후보 만들기

-  아이디어 10개 빠르게 브레인스토밍(“문제 → 대상 → 해결” 형태)
    
-  팀이 가진 자원 확인: 가능한 API/데이터, 팀의 기술 스택, 디자인 가능 여부
    
-  “너무 큰 거” 컷: 4~6주 기준 구현 불가능한 범위 제거
    

### 1-2. 선정 기준(점수표로 고르기)

각 아이디어를 1~5점으로 평가

-  문제 명확성(누가, 언제, 왜 쓰는가)
    
-  차별성(기존 서비스 대비 1~2개 포인트)
    
-  구현 가능성(기간 내 MVP 가능)
    
-  데모 임팩트(한 장면으로 가치가 보이는가)
    
-  데이터/기능 리스크(API 제한, 크롤링 난이도, 인증/결제 등)
    

### 1-3. 최종 확정 산출물

-  프로젝트 한 줄 소개(피치): “OO 사용자가 OO 상황에서 OO를 할 수 있게 돕는 서비스”
    
-  3대 핵심 기능(반드시 되는 것 3개)
    
-  성공 기준(데모에서 무엇을 보여주면 성공인지)
    

## 2) 요구사항 정의(기획서 최소 세트)

### 2-1. 사용자/시나리오

-  주요 사용자 1~2명 정의(예: 일반 사용자, 관리자)
    
-  사용자 여정(User Journey) 1개 작성(처음 방문→핵심 행동→반복 사용)
    
-  사용자 스토리 10개 작성(“사용자는 ~할 수 있어야 한다”)
    

### 2-2. 범위(Scope) 고정

-  Must / Should / Could로 기능 분류
    
-  MVP 범위 명시: “이번 기간에는 Must만”
    
-  제외 범위 명시: 결제/푸시/실시간채팅 같은 고위험 요소는 원칙적으로 제외(필요하면 대체안)
    

### 2-3. 비기능 요구사항(간단히라도)

-  로그인/권한: 없음/간단 로그인/소셜 로그인
    
-  성능 목표: 첫 화면 로딩, API 응답 목표(대략)
    
-  운영/배포: 어디에 배포할지(무료 플랜 포함)
    
-  보안/개인정보: 수집 항목 최소화, 비밀번호 저장 방식(Bcrypt 등)
    

## 3) 설계(개발 시작 가능한 수준까지)

### 3-1. 화면/플로우

-  화면 목록(페이지/모달) 나열
    
-  화면 간 이동 플로우 차트
    
-  와이어프레임(Figma 초안) 또는 손그림이라도 공유
    

### 3-2. 데이터 모델/ERD

-  핵심 엔티티 5~10개 도출
    
-  관계 정의(1:N, N:M)
    
-  ERD 작성 후 “조회/생성/수정/삭제” 시나리오로 검증
    
-  데이터 초기값/샘플 데이터 전략(Seed)
    

### 3-3. API 설계(백엔드/프론트 합의)

-  엔드포인트 목록 작성(리소스 중심)
    
-  요청/응답 DTO 스키마 정의
    
-  상태코드/에러 포맷 합의(공통 ErrorResponse)
    
-  인증 방식(JWT/세션/쿠키) 결정
    
-  페이징/정렬/필터 규칙 합의
    

### 3-4. 아키텍처/스택 확정

-  FE: 프레임워크, 상태관리, 데이터 패칭(React Query 등), UI 라이브러리
    
-  BE: 프레임워크, ORM/SQL, Validation, 문서화(Swagger)
    
-  DB: 선택(MySQL/Postgres/Oracle) + 마이그레이션 전략
    
-  배포: 프론트/백엔드/DB 각각 어디에 둘지
    

## 4) 개발 운영체계(팀프로젝트 성공의 80%)

### 4-1. Git/GitHub 규칙

-  브랜치 전략: main / develop / feature/*
    
-  커밋 메시지 규칙(예: feat/fix/refactor)
    
-  PR 규칙: 리뷰 1명 이상, 체크리스트 포함
    
-  이슈 템플릿: 기능/버그/리팩토링 구분
    

### 4-2. 작업 관리

-  칸반 보드(Backlog → Todo → In Progress → Review → Done)
    
-  태스크 쪼개기 기준: 0.5~1일 단위로 분할
    
-  데일리 스탠드업(10분): 어제/오늘/막힌 것
    
-  “막힘” 에스컬레이션 규칙: 30분 막히면 공유
    

### 4-3. 코딩 컨벤션

-  코드 포맷터/린터 적용(Prettier/ESLint, Checkstyle 등)
    
-  네이밍 규칙(폴더 구조/컴포넌트/DTO)
    
-  로그/예외 처리 규칙(서버 로그 레벨, 에러 응답 통일)
    

## 5) 품질/테스트/데모 준비

### 5-1. 테스트 전략(최소)

-  핵심 기능 3개에 대한 시나리오 테스트 문서
    
-  API 테스트(Postman 컬렉션 공유)
    
-  프론트 E2E는 어렵다면 “수동 테스트 체크리스트”라도 작성
    

### 5-2. 데모/발표

-  데모 시나리오 2~3분 구성(실패 대비 플랜 B 포함)
    
-  발표 자료 목차: 문제→해결→핵심기능→기술선정→성과/배운점
    
-  스크린샷/영상/배포 링크 준비
    

## 6) 착수 조건(“이거 되면 바로 개발 들어가도 됨”)

아래 10개가 채워지면 개발 시작 가능한 상태입니다.

-  한 줄 소개 + 핵심 기능 3개
    
-  MVP 범위(Must) 확정
    
-  화면 목록 + 플로우
    
-  ERD 초안
    
-  API 목록 초안(최소 핵심 기능)
    
-  인증/권한 방식 결정
    
-  스택 + 배포 방식 결정
    
-  Git 규칙/브랜치/PR 규칙 확정
    
-  작업 보드 생성 + 1주차 태스크 쪼개기 완료
    
-  데모 성공 기준 정의
    

## 바로 복붙해서 쓰는 “킥오프 회의 아젠다”(30~60분)

-  프로젝트 목적/기간/성공 기준 합의
    
-  아이디어 3개만 후보로 압축 → 점수표로 1개 선정
    
-  MVP Must 3개 확정
    
-  역할 분담(리드/BE/FE/디자인/PM 겸임)
    
-  다음 회의 전 숙제: 화면 목록, ERD 초안, API 초안 각 1명 담당
